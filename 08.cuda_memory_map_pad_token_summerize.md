# 08. CUDA Memory Map And Llama PAD Token Summary

## Context

Trong quá trình train/resume, có hai nhóm lỗi CUDA/OOM mới:

1. `torch.cuda.empty_cache()` báo `CUDA error: an illegal memory access was encountered`.
2. Llama bị OOM ngay lúc khởi tạo tại:

```python
self.model.resize_token_embeddings(len(self.tokenizer))
```

Lỗi thứ nhất thường là lỗi CUDA async: lỗi thật xảy ra ở kernel trước đó, nhưng chỉ được report khi gọi API CUDA sau như `empty_cache()`.

Lỗi thứ hai xảy ra vì code thêm token `"<PAD>"` mới vào tokenizer, làm Transformers phải resize embedding/lm_head của Llama. Với model 8B đang shard trên GPU gần đầy, thao tác resize này có thể gây OOM trước khi training bắt đầu.

## Files Changed

### `source/run/train.py`

Đã thêm flag mới:

```bash
--generation_max_memory_map
```

Mục đích là kiểm soát chính xác GPU nào generator được phép dùng khi `device_map=auto`.

Ví dụ khi chạy:

```bash
CUDA_VISIBLE_DEVICES=1,2,3,4
```

mapping nội bộ là:

```text
cuda:0 -> physical GPU 1
cuda:1 -> physical GPU 2
cuda:2 -> physical GPU 3
cuda:3 -> physical GPU 4
```

Nếu muốn reserve `cuda:3` cho retriever, dùng:

```bash
--generation_max_memory_map 0:7GiB,1:7GiB,2:7GiB
--retriever_device cuda:3
```

Như vậy generator chỉ được cấp memory trên `cuda:0,1,2`; retriever nằm trên `cuda:3`.

Đã thêm hàm:

```python
safe_cuda_empty_cache(context)
```

Hàm này thay cho gọi trực tiếp:

```python
torch.cuda.empty_cache()
```

Logic mới:

- gọi `torch.cuda.synchronize()` trước để lỗi CUDA async được report gần đúng vị trí hơn,
- gọi `torch.cuda.empty_cache()`,
- nếu lỗi thì in log rõ rằng lỗi thật thường xảy ra trước đó,
- gợi ý resume từ checkpoint hoặc debug ngắn với `CUDA_LAUNCH_BLOCKING=1`.

Các vị trí đã thay:

- sau validation,
- mỗi 20 train steps.

### `source/module/generate/llama.py`

Đã thêm field vào `LlamaGeneratorConfig`:

```python
max_memory_map: Optional[str] = None
```

Đã thêm parser:

```python
_parse_max_memory_map(...)
```

Hỗ trợ format:

```text
0:7GiB,1:7GiB,2:7GiB
```

hoặc có thể thêm CPU nếu cần:

```text
0:7GiB,1:7GiB,2:7GiB,cpu:32GiB
```

Khi `max_memory_map` được truyền, nó có ưu tiên cao hơn `max_memory_per_gpu`.

Đã sửa PAD token handling:

Trước đó code dùng:

```python
self.tokenizer.add_special_tokens({'pad_token': "<PAD>"})
self.model.resize_token_embeddings(len(self.tokenizer))
```

Điều này có thể gây OOM lúc load model.

Hiện tại code dùng `eos_token` làm `pad_token`:

```python
self.tokenizer.pad_token = self.tokenizer.eos_token
self.tokenizer.pad_token_id = self.tokenizer.eos_token_id
self.model.config.pad_token_id = self.tokenizer.pad_token_id
```

Không còn thêm token mới và không còn resize embedding.

## Notes

Nếu vẫn gặp illegal memory access, log mới sẽ chỉ ra rằng lỗi thật nhiều khả năng xảy ra trước `empty_cache()`.

Để debug chính xác hơn, có thể chạy một đoạn ngắn với:

```bash
CUDA_LAUNCH_BLOCKING=1
```

nhưng không nên bật lâu dài vì sẽ làm train chậm hơn.
